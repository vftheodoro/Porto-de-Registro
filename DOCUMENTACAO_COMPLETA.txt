==================================================
1) RESUMO EXECUTIVO
==================================================

O projeto e uma aplicacao Next.js (App Router) para operacao de transporte intermunicipal.
Ele concentra consulta publica (horarios, linhas, tarifas, avisos e rodoviarias) e um painel
administrativo para manutencao dos dados do sistema via arquivo data.json.

Resultado da analise:
- Estado funcional: estavel
- Fluxos principais: implementados
- Risco operacional: medio
- Maturidade tecnica: boa para operacao atual, com melhorias de seguranca recomendadas

Pontos fortes:
- Busca de horarios inteligente com suporte a conexoes
- Filtro de destinos alcancaveis por origem e tipo de dia
- Sincronizacao de estado de busca entre Home e Horarios
- Deteccao automatica de dia util/sabado/domingo/feriado no front
- Geracao de PDF operacional de horarios
- Admin com sessao assinada e rotas protegidas
- Testes automatizados cobrindo rotas criticas

Pontos de atencao:
- Sem rate-limit de tentativas de login admin
- Senha admin comparada em texto plano (sem hash)
- Arquitetura de persistencia depende de arquivo local (limite em serverless)
- Aviso de deprecacao do middleware para proxy no Next 16+

==================================================
5) FLUXO PUBLICO (PASSAGEIRO)
==================================================

Pagina Home (/):
- Mostra avisos operacionais
- Exibe busca rapida de horarios (ScheduleSearch)
- Exibe resumo de linhas/frota/servicos

Pagina Horarios (/horarios):
- Usa o mesmo componente ScheduleSearch
- UI focada em planejamento de viagem
- Atalho para pagina de rodoviarias

Componente central (ScheduleSearch):
- Busca cidades via /api/public/paradas
- Busca destinos validos via /api/public/destinos
- Executa consulta via /api/public/busca
- Exibe diretas e conexoes
- Classifica horarios como passado/proximo/futuro em tempo real
- Detecta automaticamente tipo de dia atual (inclui feriados nacionais)
- Persiste estado no localStorage (origem, destino, tipo)

Regra critica de UX/logica:
- Se houver rota direta valida, a API nao retorna conexoes artificiais

==================================================
7) APIS PUBLICAS - CONTRATOS E FUNCAO
==================================================

GET /api/public/paradas
- Retorna paradas/cidades usadas por linhas ativas
- Remove duplicidade de cidade

GET /api/public/destinos?origem&tipo
- Retorna destinos alcancaveis para origem no tipo de dia
- Usa route-planner

GET /api/public/busca?origem&destino&tipo
- Retorna:
  - resultados[] (rotas diretas)
  - conexoes[] (apenas se nao houver direto)
- Ordena e prioriza embarques mais coerentes

GET /api/public/linhas
- Lista linhas ativas com payload reduzido para listagem

GET /api/public/avisos
- Default: notificacoes operacionais
- grupo=informativos: informativos de cartoes
- grupo=todos: ambos

GET /api/public/pdf?linha_id&tipo
- Gera PDF completo para consulta/impressao
- Inclui ida/volta, observacoes, tarifas e rodape oficial

APIs:
- POST /api/admin/login
- POST /api/admin/logout
- GET /api/admin/data
- POST /api/admin/data

Autenticacao:
- Cookie admin_session
- Assinatura HMAC-SHA256 com ADMIN_SECRET
- TTL de sessao: 7 dias
- Rotas admin protegidas por middleware (exceto /admin/login)

Persistencia no admin:
- Desenvolvimento: grava data.json diretamente
- Producao/Vercel: nao grava arquivo local; retorna instrucoes para download e commit

==================================================
10) SEGURANCA - ANALISE
==================================================

Controles presentes:
- Rotas admin protegidas
- Cookie HttpOnly
- Sessao assinada

Riscos identificados:
- Sem rate limit no login
- Sem bloqueio progressivo por tentativas
- Senha em texto plano (sem hash)
- Sem auditoria persistente de alteracoes (quem alterou o que)

Recomendacoes:
1. Adicionar rate limit em /api/admin/login.
2. Migrar para hash de senha (bcrypt/argon2) e comparacao segura.
3. Registrar historico de alteracoes do data.json.
4. Migrar middleware para proxy conforme recomendacao do Next.
